프로젝트 개발 프로세스

⏹️ 소프트웨어 개발 방법론(복습)

1. 폭포수 모델

  • 요구사항 분석, 설계, 구현, 테스트, 유지보수의 단계를 순차적으로 진행
  • 각 단계가 확실하게 분리되어 있어 이해하기 쉬움
  • 단계가 끝나야 다음 단계로 진행 가능

2. V-Model

  • 폭포수 모델의 단점을 보완한 모델
  • 각 단계의 검증 및 검토를 병행하여 진행
  • 개발 단계와 테스트 단계가 대응되는 V자 모양의 구조

3. 애자일 방법론

  • 요구사항이 빈번하게 변하는 프로젝트에 적합
  • 짧은 주기의 스프린트를 통해 반복적이고 점진적인 개발
  • 사용자의 피드백을 적극 수용하여 유연하게 대응

⏹️주요 개발 단계

[요구사항 분석] -> [설계] -> [구현] -> [테스트] -> [유지보수] 순으로 진행되는 소프트웨어 개발 프로세스를 기본으로 하여 설명을 드리겠습니다.

프로젝트의 성질에 따라 다양한 방법론과 접근 방식이 존재하며, 이를 적절히 조합하여 프로젝트를 진행하는 것이 중요합니다.

1. 요구사항 분석

  • 사용자의 요구사항을 수집하고 분석
  • 요구사항 명세서 작성
  • 요구사항의 우선순위와 변경 관리
  • 요구사항 검증

2. 설계

시스템 아키텍처 설계

모노리틱 아키텍처
  • 전통적인 아키텍처로, 단일 애플리케이션으로 구성
  • 개발과 배포가 간단하나 유지보수가 어려울 수 있음

ex) Unity, Unreal Engine의 경우 모노리틱 아키텍처를 사용하여 게임 엔진을 개발하고 있습니다. 게임 엔진은 게임의 모든 기능을 담고 있으며, 게임의 실행, 렌더링, 물리 시뮬레이션, 네트워킹 등의 기능을 제공합니다.

마이크로서비스 아키텍처
  • 여러 개의 작은 서비스로 구성
  • 각 서비스는 독립적으로 배포 및 확장 가능

ex) AWS, Azure, Google Cloud 등의 클라우드 서비스는 마이크로서비스 아키텍처를 사용하여 서비스를 제공하고 있습니다. 각 서비스는 독립적으로 운영되며, 서비스 간의 통신을 통해 서비스를 제공합니다.

데이터베이스 설계

  • 데이터베이스 설계는 데이터베이스의 구조와 관계를 정의하는 과정으로, 데이터베이스의 성능과 효율성을 고려하여 설계합니다.
  • 관계형 데이터베이스와 비관계형 데이터베이스로 나눌 수 있습니다.
  • 데이터베이스 설계는 데이터베이스의 정규화, 인덱싱, 파티셔닝 등을 고려하여 최적화합니다.
    • 정규화: 데이터 중복을 최소화하여 데이터 일관성을 유지
    • 인덱싱: 데이터 검색 속도를 향상
    • 파티셔닝: 대용량 데이터 처리를 위한 데이터 분할
    • 샤딩: 데이터베이스의 부하 분산을 위한 데이터 분할
관계형 데이터베이스
  • 테이블 간의 관계를 정의하여 데이터를 저장
  • SQL을 사용하여 데이터를 조회하고 조작
  • MySQL, PostgreSQL, Oracle 등

싱글플레이 콘텐츠 구현시 SQLite를 사용하는 경우가 많습니다. SQLite는 경량 데이터베이스로, 단일 파일로 데이터를 저장하며, 별도의 서버가 필요하지 않습니다.

Unity, Unreal Engine 등의 게임 엔진에서도 SQLite를 지원하고 있습니다. 에셋 스토어나 Fab에서 SQLite 관련 플러그인을 다운로드하여 사용하면 더욱 손쉽게 데이터베이스를 구현하여 사용할 수 있습니다.

Unity SQLite 사용 예시
using System.Data;
using Mono.Data.Sqlite;

public class SQLiteTest : MonoBehaviour
{
    private string connectionString;

    void Start()
    {
        connectionString = "URI=file:Assets/StreamingAssets/test.db";
        using (var connection = new SqliteConnection(connectionString))
        {
            connection.Open();

            using (var command = connection.CreateCommand())
            {
                command.CommandText = "CREATE TABLE IF NOT EXISTS test (id INTEGER PRIMARY KEY, name TEXT)";
                command.ExecuteNonQuery();
            }

            using (var command = connection.CreateCommand())
            {
                command.CommandText = "INSERT INTO test (name) VALUES ('홍길동')";
                command.ExecuteNonQuery();
            }

            using (var command = connection.CreateCommand())
            {
                command.CommandText = "SELECT * FROM test";
                using (var reader = command.ExecuteReader())
                {
                    while (reader.Read())
                    {
                        Debug.Log(reader["name"]);
                    }
                }
            }
        }
    }
}
Unreal Engine SQLite 사용 예시
  • BP 방식
    • Fab에서 SQLite 플러그인 설치 (프로젝트에 알맞는 플러그인 선택)
  • C++ 방식

#include "SQLiteCpp/SQLiteCpp.h"

void SQLiteTest()
{
    SQLite::Database db("test.db", SQLite::OPEN_READWRITE | SQLite::OPEN_CREATE);

    db.exec("CREATE TABLE IF NOT EXISTS test (id INTEGER PRIMARY KEY, name TEXT)");
    db.exec("INSERT INTO test (name) VALUES ('홍길동')");

    SQLite::Statement query(db, "SELECT * FROM test");
    while (query.executeStep())
    {
        std::string name = query.getColumn("name");
        UE_LOG(LogTemp, Warning, TEXT("%s"), *FString(name.c_str()));
    }
}
비관계형 데이터베이스
  • 테이블 간의 관계를 정의하지 않고 데이터를 저장
  • JSON, XML 등의 형식으로 데이터를 저장
  • MongoDB, Redis, Cassandra 등

대용량의 데이터를 저장하고 관리하는데 적합한 NoSQL 데이터베이스로, 빠른 데이터 처리와 확장성을 제공합니다.

Unity MongoDB 사용 예시

using MongoDB.Driver;

public class MongoDBTest : MonoBehaviour
{
    private IMongoCollection<BsonDocument> collection;

    void Start()
    {
        var client = new MongoClient("mongodb://localhost:27017");
        var database = client.GetDatabase("test");
        collection = database.GetCollection<BsonDocument>("test");

        var document = new BsonDocument
        {
            { "name", "홍길동" }
        };
        collection.InsertOne(document);

        var filter = Builders<BsonDocument>.Filter.Empty;
        var result = collection.Find(filter).ToList();
        foreach (var item in result)
        {
            Debug.Log(item["name"]);
        }
    }
}
Unreal Engine MongoDB 사용 예시
  • BP 방식
    • Fab에서 MongoDB 플러그인 설치 (프로젝트에 알맞는 플러그인 선택)
  • C++ 방식

#include "mongocxx/client.hpp"
#include "mongocxx/instance.hpp"
#include "mongocxx/uri.hpp"

void MongoDBTest()
{
    mongocxx::instance instance{};
    mongocxx::client client{mongocxx::uri{"mongodb://localhost:27017"}};
    auto collection = client["test"]["test"];

    auto document = bsoncxx::builder::stream::document{} << "name" << "홍길동" << bsoncxx::builder::stream::finalize;
    collection.insert_one(document.view());

    auto cursor = collection.find({});
    for (auto&& doc : cursor)
    {
        auto view = doc.view();
        UE_LOG(LogTemp, Warning, TEXT("%s"), *FString(view["name"].get_utf8().value.to_string().c_str()));
    }
}

3. 구현

프로그래밍

단순히 코드를 작성하는 것이 아닌, 프로젝트에 맞는 구조적이고 효율적인 코드를 작성하는 것이 중요합니다. 코드는 지향하는 목표에 따라 개발 프로그래밍 패러다임이 달라질 수 있습니다. 근래에는 객체지향 뿐만 아니라 함수형 프로그래밍, 리액티브 프로그래밍 등 다양한 패러다임이 등장하고 있습니다.

프로그래밍 패러다임 종류

프로그래밍 패러다임은 프로그래밍 언어의 설계와 구현 방식을 결정하는 기본 철학이자 방법론입니다. 프로그래밍 패러다임은 프로그래밍 언어의 특성과 기능을 결정하며, 프로그래밍 언어의 종류에 따라 다양한 패러다임이 존재합니다.

  • 절차지향 프로그래밍(Procedural Programming)
    • 명령어들의 집합으로 프로그램을 구성
    • C언어, 파이썬 등
    • 코드의 순차적인 실행을 중요시함
  • 객체지향 프로그래밍(Object-Oriented Programming)
    • 실감 미디어 콘텐츠 구현시 가장 많이 사용되는 패러다임.(Unity, Unreal Engine)
    • 객체를 중심으로 프로그램을 구성
    • Java, C++, Python 등
    • 코드의 재사용성과 유지보수성을 중요시함
  • 함수형 프로그래밍(Functional Programming)
    • 함수를 일급 객체로 취급
    • Haskell, Scala, Clojure 등
    • 부작용을 최소화하고 불변성을 유지
  • 리액티브 프로그래밍(Reactive Programming)
    • 데이터 스트림과 변경사항 전파에 중점
    • RxJava, RxJS, Reactor 등
    • 비동기적인 이벤트 처리에 적합
      • Unity에서는 UniRx를 사용하여 리액티브 프로그래밍을 구현할 수 있습니다.
      • Unreal Engine에서는 RxCpp를 사용하여 리액티브 프로그래밍을 구현할 수 있습니다.
      • 특히 게임 엔진에서는 이벤트 처리와 상태 변화를 효율적으로 관리하기 위해 리액티브 프로그래밍을 활용하는 경우가 많습니다.
  • DOTS (Data-Oriented Technology Stack)
    • 데이터 지향 프로그래밍 패러다임
    • ECS(Entity-Component-System) 아키텍처를 기반으로 함
    • 대용량 데이터 처리와 최적화에 특화
  • 기타 패러다임
    • 머신러닝, 빅데이터, IoT 등 다양한 분야에 따라 다양한 패러다임이 존재
디자인 패턴을 활용한 프로그래밍

디자인 패턴이란 프로그래밍에서 자주 발생하는 문제를 해결하기 위한 방법 중 하나로, 공통된 설계 문제를 해결하기 위한 해결책을 제시한 것.

생성 패턴(Creational Patterns)
  • 객체 생성에 관련된 패턴
  • 객체 생성 방식을 추상화하여 유연성을 확보
  • 예: Singleton, Factory Method, Abstract Factory
    • Factory Method
      • 객체 생성을 서브 클래스로 분리하여 확장성을 향상
      • 객체 생성 방식을 변경하더라도 클라이언트 코드에 영향을 주지 않도록 함
    • Abstract Factory
      • 관련 객체들의 생성을 추상화하여 객체 생성을 위한 인터페이스 제공
      • 객체의 생성 방식을 변경하더라도 클라이언트 코드에 영향을 주지 않도록 함
    • Singleton
      • 시스템 전체에서 단일 객체 인스턴스를 유지
      • 객체의 생성과 관리를 중앙에서 담당
구조 패턴(Structural Patterns)
  • 객체와 클래스의 구조를 설계하는 패턴
  • 클래스 간의 관계를 조직화하여 유지보수성을 향상
  • 예: Adapter, Composite, Decorator
    • Adapter
      • 호환되지 않는 인터페이스를 함께 동작할 수 있도록 변환
      • 클라이언트 코드의 수정 없이 호환성을 확보
    • Composite
      • 객체들을 트리 구조로 구성하여 계층적인 구조를 표현
      • 단일 객체와 복합 객체를 동일하게 다룰 수 있도록 함
    • Decorator
      • 객체에 동적으로 새로운 책임을 추가
      • 상속을 통해 기능을 확장하는 것보다 유연하게 기능 추가 가능
행위 패턴(Behavioral Patterns)
  • 객체 간의 상호작용과 책임 분배에 관련된 패턴
  • 객체 간의 효율적인 소통을 위한 방법 제시
  • 예: Observer, Strategy, Template Method
    • Observer
      • 객체 간의 일대다 의존 관계를 정의
      • 상태 변화 시 관련 객체들에게 알림을 전달
    • Strategy
      • 알고리즘을 캡슐화하여 동적으로 교체 가능
      • 알고리즘을 사용하는 클라이언트 코드에 영향을 주지 않도록 함
    • Template Method
      • 알고리즘의 구조를 정의하고 일부 단계를 서브 클래스로 분리
      • 알고리즘의 일부 단계를 서브 클래스에서 재정의하여 확장성을 향상
기타 패턴
  • 위 패턴 이외에도 다양한 디자인 패턴이 존재하기 때문에 프로젝트의 특성에 따라 적합한 패턴을 선택하는 것이 중요

4. 테스트

테스트 방법론

소프트웨어 테스트는 소프트웨어의 품질을 검증하고, 사용자의 요구사항을 충족시키는지 확인하는 과정입니다. 테스트 방법론은 테스트를 어떻게 수행할지에 대한 체계적인 방법을 제공합니다.

테스트 방법론은 소프트웨어의 특성과 목적에 따라 다양하게 적용될 수 있습니다. 소프트웨어의 특성에 따라 수행되는 테스트 방법론은 크게 기능 테스트와 비기능 테스트로 나눌 수 있습니다.

기능 테스트

기능 테스트는 소프트웨어의 기능을 검증하는 테스트로, 사용자의 요구사항을 충족하는지 확인하는 과정입니다. 기능 테스트는 소프트웨어의 기능을 테스트하는 단계별 테스트와 전체 시스템을 테스트하는 종합적인 테스트로 나눌 수 있습니다. 성능이나 보안 등의 테스트는 비기능 테스트에서 진행 됩니다.

  • 단위 테스트(Unit Test)
    • 소프트웨어의 최소 단위인 모듈을 테스트
    • 모듈의 기능이 정상적으로 동작하는지 확인
    • JUnit, NUnit, PyTest 등의 테스트 프레임워크 활용
  • 통합 테스트(Integration Test)
    • 모듈들이 결합된 시스템의 동작을 테스트
    • 모듈 간의 인터페이스와 상호작용을 확인
    • 모듈 간의 통합이 정상적으로 이루어지는지 확인
  • 시스템 테스트(System Test)
    • 전체 시스템의 동작을 테스트
    • 사용자의 요구사항을 충족하는지 확인
    • 시스템의 기능, 성능, 보안 등을 종합적으로 테스트
  • 인수 테스트(Acceptance Test)
    • 사용자의 요구사항을 검증하는 테스트
    • 사용자의 입장에서 시스템을 테스트
    • 사용자의 요구사항을 충족하는지 확인
비기능 테스트

비기능 테스트는 소프트웨어의 성능, 안정성, 보안 등의 비기능적인 요구사항을 검증하는 테스트입니다. 비기능 테스트는 사용자의 요구사항을 충족시키는 것뿐만 아니라, 소프트웨어의 품질을 보장하기 위해 수행됩니다.

  • 성능 테스트(Performance Test)
    • 시스템의 성능을 테스트
    • 응답 시간, 처리량, 부하 테스트 등을 수행
    • 시스템의 성능을 최적화하고 안정성을 검증
  • 부하 테스트(Stress Test)
    • 시스템의 한계를 테스트
    • 시스템이 수용할 수 있는 부하를 확인
    • 시스템의 한계를 확인하고 대응 방안을 모색
  • 보안 테스트(Security Test)
    • 시스템의 보안을 테스트
    • 시스템의 취약점을 확인하고 보완
    • 시스템의 보안을 강화하고 사용자 정보를 보호

5. 🔧 유지보수

유지보수는 소프트웨어가 배포된 후 시스템의 기능을 유지하고 개선하는 과정입니다. 이는 버그 수정, 성능 개선, 새로운 기능 추가 등을 포함합니다.

유지보수의 유형

  1. 수정 유지보수 (Corrective Maintenance)

    • 시스템 운영 중 발견된 오류나 버그를 수정
    • 사용자 보고나 모니터링을 통해 식별된 문제 해결

  2. 적응 유지보수 (Adaptive Maintenance)

    • 변화하는 환경(하드웨어, 운영체제 등)에 시스템을 적응시킴
    • 새로운 기술이나 플랫폼에 대응하기 위한 업데이트

  3. 완전 유지보수 (Perfective Maintenance)

    • 시스템의 성능을 개선하거나 새로운 기능을 추가
    • 사용자 요구사항 변화에 따른 기능 개선

  4. 예방 유지보수 (Preventive Maintenance)

    • 잠재적인 문제를 사전에 식별하고 예방
    • 코드 리팩토링, 문서화 개선 등을 통한 시스템 품질 향상

효과적인 유지보수 전략

  1. 문서화

    • 시스템 구조, 코드, 변경 이력 등을 철저히 문서화
    • 새로운 개발자도 쉽게 이해할 수 있도록 명확한 문서 작성

  2. 버전 관리

    • Git 등의 버전 관리 시스템을 활용하여 코드 변경 이력 관리
    • 브랜치 전략을 통한 효율적인 개발 및 배포 관리

  3. 자동화된 테스트

    • 단위 테스트, 통합 테스트 등을 자동화하여 변경 사항의 영향을 빠르게 확인
    • 지속적 통합(CI) 도구를 활용한 자동 테스트 실행

  4. 모니터링 및 로깅

    • 시스템 성능 및 오류를 실시간으로 모니터링
    • 로그 분석을 통한 문제 원인 파악 및 예방
    • Unity에서는 Unity Analytics, Unreal Engine에서는 Unreal Insights 등의 모니터링 도구를 활용할 수 있습니다.
    • 서버 기반 게임의 경우, 서버 모니터링 도구를 활용하여 서버의 상태를 실시간으로 확인할 수 있습니다.

  5. 사용자 피드백 관리

    • 사용자로부터 지속적인 피드백 수집 및 분석
    • 우선순위를 설정하여 중요한 개선사항 먼저 반영

  6. 정기적인 코드 리뷰

    • 팀 내 정기적인 코드 리뷰를 통한 코드 품질 유지
    • 베스트 프랙티스 공유 및 지식 전파

유지보수 도구

  • 이슈 트래킹 시스템: Jira, GitHub Issues, Trello 등
  • 모니터링 도구: Prometheus, Grafana, New Relic 등
  • 로그 관리: ELK Stack (Elasticsearch, Logstash, Kibana), Splunk 등
  • 성능 분석: JProfiler, YourKit, VisualVM 등

👩‍💻/👨‍💻 페어 프로그래밍

두 명의 개발자가 하나의 컴퓨터에서 작업하는 방식으로, 코드 리뷰와 피드백을 실시간으로 주고받을 수 있습니다. 페어 프로그래밍은 코드 품질을 높이고, 버그를 줄이며, 지식 공유를 촉진하는 데 효과적입니다.

  • 먼저 코드를 작성하는 사람은 '드라이버', 코드를 검토하고 피드백을 주는 사람은 '네비게이터'로 역할을 나눕니다.
  • 일정 주기로 역할을 교체하여 서로의 역할을 경험하고 지식을 공유합니다.
  • 주의할 점은 서로의 의견을 존중하고, 피드백을 건설적으로 주고받아야 합니다.
  • 장점
    • 코드 품질 향상
    • 버그 발견 및 수정 용이
    • 지식 공유 및 팀원 간 의사소통 촉진
  • 단점
    • 의견 충돌이 발생할 수 있음
    • 개발자간 커뮤니케이션 부담
    • 피로도가 상당함.

🔍 코드 리뷰

코드 리뷰는 개발자들이 작성한 코드를 체계적으로 검토하고 피드백을 주고받는 중요한 프로세스입니다. 이를 통해 코드 품질을 향상시키고, 버그를 조기에 발견하며, 팀 내 지식 공유를 촉진할 수 있습니다.

코드 리뷰의 주요 목적

  1. 코드 품질 향상
  2. 버그 조기 발견 및 수정
  3. 개발자 간 지식 공유
  4. 일관된 코딩 스타일 유지
  5. 보안 취약점 식별

코드 리뷰 프로세스

  1. 코드 작성: 개발자가 기능을 구현하고 코드를 작성합니다.
  2. 리뷰 요청: 작성된 코드에 대해 리뷰를 요청합니다.
  3. 코드 검토: 리뷰어가 코드를 검토하고 피드백을 제공합니다.
  4. 피드백 반영: 코드 작성자가 받은 피드백을 바탕으로 코드를 수정합니다.
  5. 재검토: 수정된 코드를 재검토하고 필요시 추가 수정을 진행합니다.
  6. 승인: 모든 문제가 해결되면 코드를 승인하고 병합합니다.
코드 리뷰 예시
  • GitHub Pull Request
    • 코드 변경 사항을 비교하고 인라인 코멘트를 달 수 있습니다.
    • 코드 변경 이력과 토론 내용을 한눈에 확인할 수 있습니다.
    • 코드 리뷰어가 코드 변경 사항을 승인하거나 요청 사항을 제시할 수 있습니다.

효과적인 코드 리뷰 방법

  1. 작은 단위로 리뷰: 한 번에 200-400줄 정도의 코드를 리뷰하는 것이 가장 효과적입니다.
  2. 체크리스트 사용: 일관된 리뷰를 위해 체크리스트를 활용합니다.
  3. 건설적인 피드백: 문제점뿐만 아니라 개선 방안도 함께 제시합니다.
  4. 코드의 맥락 이해: 코드의 목적과 전체적인 맥락을 고려하여 리뷰합니다.
  5. 자동화 도구 활용: 정적 분석 도구를 사용하여 기본적인 문제를 자동으로 검출합니다.

코드 리뷰 도구

  1. GitHub Pull Requests: 변경사항을 쉽게 비교하고 인라인 코멘트를 달 수 있습니다.
  2. GitLab Merge Requests: GitHub와 유사한 기능을 제공합니다.
  3. Gerrit: 웹 기반의 코드 리뷰 시스템으로, 복잡한 워크플로우를 지원합니다.
  4. Phabricator: 페이스북에서 개발한 오픈소스 코드 리뷰 도구입니다.

코드 리뷰 모범 사례

  1. 존중과 겸손: 코드가 아닌 사람을 비판하지 않습니다.
  2. 명확한 설명: 왜 변경이 필요한지 명확히 설명합니다.
  3. 우선순위 설정: 중요한 문제에 집중합니다.
  4. 긍정적인 피드백: 좋은 점도 언급하여 동기를 부여합니다.
  5. 학습 기회로 활용: 코드 리뷰를 통해 서로 배우고 성장합니다.

스크럼 미팅

스크럼 미팅은 스크럼 프레임워크에서 정기적으로 진행되는 여러 유형의 회의를 총칭합니다.

이러한 미팅들은 팀의 진행 상황을 추적하고, 문제를 해결하며, 지속적인 개선을 위한 기회를 제공합니다.

주요 스크럼 미팅 유형

1. 일일 스크럼 (Daily Scrum)
  • 목적: 팀의 진행 상황을 공유하고 장애물을 식별합니다.
  • 주기: 매일 같은 시간, 같은 장소에서 진행됩니다.
  • 시간: 15분을 넘지 않도록 합니다.
  • 참석자: 개발 팀 멤버들이 참석합니다.
  • 진행 방식: 각 팀원이 다음 세 가지 질문에 답변합니다:
    1. 어제 무엇을 했는가?
    2. 오늘 무엇을 할 것인가?
    3. 진행에 방해가 되는 장애물이 있는가?
2. 스프린트 계획 미팅 (Sprint Planning)
  • 목적: 다음 스프린트에서 수행할 작업을 계획합니다.
  • 주기: 각 스프린트 시작 시 진행됩니다.
  • 시간: 보통 4~8시간 정도 소요됩니다 (2주 스프린트 기준).
  • 참석자: 전체 스크럼 팀이 참석합니다.
  • 주요 활동:
    • 스프린트 목표 설정
    • 제품 백로그에서 스프린트에 포함할 항목 선택
    • 선택된 항목에 대한 작업 계획 수립
3. 스프린트 리뷰 (Sprint Review)
  • 목적: 완료된 작업을 검토하고 피드백을 받습니다.
  • 주기: 각 스프린트 종료 시 진행됩니다.
  • 시간: 보통 4시간 이내로 진행됩니다 (2주 스프린트 기준).
  • 참석자: 스크럼 팀과 주요 이해관계자들이 참석합니다.
  • 주요 활동:
    • 완료된 작업 시연
    • 피드백 수집
    • 제품 백로그 업데이트
4. 스프린트 회고 (Sprint Retrospective)
  • 목적: 지난 스프린트를 되돌아보고 개선점을 찾습니다.
  • 주기: 스프린트 리뷰 후, 다음 스프린트 시작 전에 진행됩니다.
  • 시간: 보통 1~3시간 정도 소요됩니다.
  • 참석자: 스크럼 팀 전체가 참석합니다.
  • 주요 활동:
    • 지난 스프린트에서 잘된 점과 개선이 필요한 점 논의
    • 개선 계획 수립
  • 진행 순서:
    1. 성공 요인: 지난 스프린트에서 성공한 요인을 공유합니다.
    2. 개선 사항: 개선이 필요한 부분을 식별하고 논의합니다.
    3. 향후 계획: 개선 사항을 바탕으로 향후 계획을 수립합니다.

스크럼 미팅의 효과적인 진행 방법

  1. 시간 준수: 각 미팅의 시간 제한을 엄격히 지켜 효율성을 높입니다.
  2. 포커스 유지: 미팅의 목적에 집중하고 불필요한 논의는 피합니다.
  3. 적극적인 참여 독려: 모든 팀원이 의견을 공유할 수 있도록 합니다.
  4. 시각화 도구 활용: 칸반 보드나 번다운 차트 등을 사용해 진행 상황을 시각화합니다.
  5. 액션 아이템 추적: 미팅에서 나온 결정사항과 할 일을 명확히 기록하고 추적합니다.

🧠 워크샵

워크샵은 팀원들이 함께 모여 특정 주제에 대해 집중적으로 학습하고, 문제를 해결하며, 아이디어를 공유하는 협업 세션입니다.

효과적인 워크샵은 팀의 역량을 강화하고 프로젝트의 성공 가능성을 높일 수 있습니다.

워크샵의 주요 목적

  • 지식 공유 및 학습
  • 문제 해결 및 의사결정
  • 팀 빌딩 및 협업 강화
  • 창의성 촉진
  • 프로젝트 계획 수립 및 검토

워크샵 유형

  1. 기술 워크샵: 새로운 기술이나 도구에 대한 학습
  2. 아이디어 발상 워크샵: 브레인스토밍을 통한 창의적 해결책 도출
  3. 프로젝트 킥오프 워크샵: 프로젝트 시작 시 목표 설정 및 계획 수립
  4. 회고 워크샵: 프로젝트나 스프린트 종료 후 성과 분석 및 개선점 도출
  5. 팀 빌딩 워크샵: 팀워크 강화 및 의사소통 개선

효과적인 워크샵 진행 방법

  1. 명확한 목표 설정: 워크샵의 목적과 기대 결과를 명확히 합니다.
  2. 적절한 참가자 선정: 주제와 관련된 핵심 인원들을 참여시킵니다.
  3. 사전 준비: 필요한 자료와 도구를 미리 준비합니다.
  4. 다양한 활동 구성: 강의, 토론, 실습 등을 적절히 혼합합니다.
  5. 시간 관리: 각 세션에 적절한 시간을 배분하고 준수합니다.
  6. 참여 독려: 모든 참가자가 적극적으로 참여할 수 있도록 유도합니다.
  7. 결과 정리 및 공유: 워크샵 결과를 문서화하고 팀과 공유합니다.

워크샵 도구 및 기법

  • 화이트보드 / 플립차트: 아이디어를 시각화하고 정리하는 데 활용
  • 포스트잇: 개별 아이디어를 쉽게 정리하고 그룹화하는 데 사용
  • 마인드맵: 복잡한 아이디어나 개념을 구조화하는 데 효과적
  • SWOT 분석: 프로젝트나 팀의 강점, 약점, 기회, 위협을 분석
  • 디자인 씽킹: 사용자 중심의 문제 해결 방법론
  • 월드 카페: 다양한 주제에 대해 소그룹 토론을 진행하는 방식

온라인 워크샵

  • 온라인 워크샵은 지리적 제약을 극복하고, 유연한 시간대에서 협업할 수 있는 장점이 있습니다.
  • 주의할 점은 온라인 환경에서의 참여자들의 집중도 유지와 기술적인 문제 해결이 필요합니다.
  • Zoom, Google Meet, Microsoft Teams 등의 온라인 회의 도구를 활용하여 워크샵을 진행할 수 있습니다.

실습 활동

⏹️프로젝트 계획 수립

1. 프로젝트 목표 설정

  • SMART 기법을 사용하여 프로젝트 목표를 구체화합니다:
    • Specific (구체적)
    • Measurable (측정 가능한)
    • Achievable (달성 가능한)
    • Relevant (관련성 있는)
    • Time-bound (기한이 있는)
  • 프로젝트의 목표와 범위를 명확히 정의합니다.

2. 요구사항 명세서 작성

  • 요구사항 명세서를 작성합니다. 이때 요구사항은 다음과 같은 내용을 포함합니다:
    (기능적 요구사항과 비기능적 요구사항을 구분하여 작성합니다.)
    • 기능적 요구사항
    • 비기능적 요구사항
    • 우선순위 설정
    • 간단한 설명
  • 예시1) 아이템 시스템 구현
    • 기능적 요구사항
      • 아이템 생성, 수정, 삭제
      • 아이템 구매, 판매
      • 아이템 보유 및 장착
    • 비기능적 요구사항
      • UI 응답 시간 0.5초 이내
      • 아이템 조작을 막을 수 있는 보안 기능
      • 컴퓨터 사양에 영향을 최소화 하는 최적화
  • 예시2) 전투 시스템 구현
    • 기능적 요구사항
      • 캐릭터 전투 시스템 구현
      • 캐릭터 스킬 시스템 구현
      • 캐릭터 상태 이상 시스템 구현
    • 비기능적 요구사항
      • 전투 시스템의 안정성 보장
      • 전투 시스템의 성능 최적화
      • 전투 시스템의 확장성 고려

⏹️개발 방법론 선택 및 적용

1. 개발 방법론 비교 분석

  • 폭포수 모델, V-모델, 애자일 방법론 중 프로젝트에 적합한 방법론을 선택합니다.
  • 선택한 방법론의 장단점을 분석하고, 프로젝트에 어떻게 적용할지 계획을 수립합니다.

2. 스프린트 계획 수립 (애자일 방법론 선택 시)

  • 전체 프로젝트 기간을 2-3개의 스프린트로 나눕니다.
  • 각 스프린트의 목표와 완료해야 할 작업 목록을 작성합니다.

⏹️협업 도구 활용

1. 프로젝트 관리 도구 설정

  • Trello, Jira, 또는 Asana 중 하나를 선택하여 프로젝트 보드를 생성합니다.
  • 작업 항목을 "To Do", "In Progress", "Done" 등의 열로 구분하여 등록합니다.

2. 버전 관리 시스템 설정

  • GitHub 또는 GitLab에 프로젝트 저장소를 생성합니다.
  • 팀원들을 저장소에 초대하고, 브랜치 전략을 수립합니다.

3. 커뮤니케이션 채널 구축

  • Slack 또는 Discord에 프로젝트 전용 채널을 만들고 팀원들을 초대합니다.
  • 주요 공지사항, 일일 스크럼 미팅, 코드 리뷰 등을 위한 세부 채널을 구성합니다.

⏹️팀 워크샵 진행

진행상황 공유 및 피드백

  • 지금까지 진행한 상황을 공유하고, 피드백을 주고받습니다.
  • 프로젝트 목표와 요구사항 명세서를 검토하고, 필요한 수정사항을 논의합니다.
  • 개발 방법론과 스프린트 계획을 공유하고, 팀원들의 의견을 수렴합니다.

작업 분배 및 계획 수립

  • 각 팀원들에게 작업을 분배하고, 스프린트 계획을 수립합니다.
  • 각 팀원들이 스프린트 목표를 이해하고, 작업을 시작할 수 있도록 지원합니다.
  • 작업의 우선순위를 설정하고, 각 작업의 예상 소요 시간을 고려하여 계획을 수립합니다.
